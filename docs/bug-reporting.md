# Bug 报告指南

[English version](en/bug-reporting.md)

感谢你帮助改进 Apache LDAP API。高质量的 bug 报告应当让维护者能快速判断影响范围，并能在本地复现问题。

## 报告前

- 搜索现有 Issue，确认问题尚未被报告或已有修复计划。
- 尽量使用当前维护版本或最新 `master` 验证问题是否仍然存在。
- 准备最小复现代码、测试、LDIF、Schema、LDAP 请求或 Gradle 命令。
- 移除日志中的凭据、内部地址、生产数据和用户隐私。

未修复的安全漏洞不要通过公开 Issue 报告。请阅读 `../SECURITY.md` 并优先使用 GitHub 私有安全报告。

## Issue 应包含的信息

- 项目版本、依赖坐标或提交哈希。
- 受影响模块，例如 LDAP model、codec、client API、Schema、DSML、OSGi 或 Gradle 构建。
- 使用路径，例如直接库调用、LDAP 客户端连接、编解码、Schema/LDIF 处理或框架集成。
- JDK、操作系统、Gradle、LDAP 服务端、OSGi 容器等运行环境。
- 期望行为和实际行为。
- 最小复现方式，最好是能加入仓库的测试用例。
- 相关异常堆栈、构建日志或协议数据片段。
- 已尝试过的规避方式或排查结论。

## 最小复现建议

- 只保留触发问题所需的依赖、输入数据和代码路径。
- 对解析、Schema、LDIF、DN/RDN、过滤器等问题，尽量提供完整输入文本。
- 对客户端或网络问题，说明 LDAP 服务端类型、连接参数类别、认证方式和触发操作，但不要公开密码或令牌。
- 对并发或时序问题，说明线程数、循环次数、超时设置和失败概率。
- 对构建问题，提供完整 Gradle 命令和最小相关日志。

## 适合公开 Issue 的内容

- 可复现的功能缺陷、解析错误、兼容性问题或构建失败。
- 文档缺失、示例错误或说明不清。
- 经过说明的功能请求或改进建议。

## 不适合公开 Issue 的内容

- 数据泄露、任意文件访问、可利用反序列化、权限绕过或可远程触发拒绝服务等安全问题的细节。
- 私有凭据、内部地址、生产目录数据或用户隐私。
- 与本项目无关的通用 LDAP 运维问题。

## 修复协作

如果你愿意提交修复，请先阅读 `../CONTRIBUTING.md`。修复 bug 时请尽量补充能复现问题的测试，并在 Pull Request 中说明运行过的 Gradle 任务。
