# 安全策略

[English version](docs/en/SECURITY.md)

## 支持范围

当前 `master` 分支是主要维护线。安全修复会优先进入 `master`，是否回移到历史版本取决于影响范围、风险和维护成本。

## 报告漏洞

请不要在公开 Issue 中披露未修复的安全漏洞细节。

推荐流程：

1. 使用 GitHub 的私有安全报告能力提交漏洞报告。
2. 如果当前仓库没有启用私有安全报告，请创建一个不包含漏洞细节的公开 Issue，请维护者提供私下沟通方式。
3. 报告中尽量包含受影响版本、触发条件、影响范围、复现步骤、临时缓解建议和你愿意公开的署名信息。

## 安全敏感范围

以下问题请按安全漏洞处理，并在协调披露前避免公开细节：

- LDAP 条目、凭据、配置或日志的非预期数据暴露。
- 由项目代码导致的任意文件访问或路径穿越。
- 可跨越信任边界的反序列化、解析器或编解码可利用行为。
- 支持的客户端或集成路径中的认证、授权或权限绕过。
- 具有实际影响的可远程触发拒绝服务。
- 通过本项目受支持使用路径可达、可利用的依赖漏洞。

## 响应预期

维护者会尽力尽快确认报告、评估影响，并在有修复或缓解方案后协调披露。安全问题在确认前可能需要多轮复现和影响面分析。

## 范围说明

以下内容通常不视为安全漏洞：

- 仅影响本地测试环境的构建失败。
- 已公开依赖漏洞但本项目不可达、不可利用的路径。
- 需要完全受信任调用方主动执行恶意代码的场景。

如果不确定是否属于漏洞，请按安全问题处理并优先私下报告。
